# LiNGAM × Utility Causality（GEM-RAG的因果性）— 比較・拡張ブレスト

## 0. 前提の整理

### LiNGAMファミリーの全体像

| 手法 | 対象 | 仮定 | 出力 |
|---|---|---|---|
| **ICA-LiNGAM** | 横断データ | 線形, 非ガウス誤差, DAG, 潜在共通原因なし | 完全因果DAG |
| **DirectLiNGAM** | 横断データ | 同上（より安定） | 完全因果DAG + 因果順序 |
| **VARLiNGAM** | 時系列 | 線形, 非ガウス, 同時点DAG, 潜在共通原因なし | 時間遅れ + 同時点の因果DAG |
| **RCD** | 横断データ | 線形, 非ガウス, DAG（潜在共通原因あり） | 因果DAG + 双方向弧（潜在共通原因） |
| **CAM-UV** | 横断データ | **非線形（GAM）**, DAG, 潜在変数あり | 因果DAG + 未観測パスの検出 |
| **Longitudinal LiNGAM**（Okuda+ 2025） | 縦断データ | 線形, 非ガウス, ワークフロー制約 | 時間内+時間間因果構造 |

### Utility Causality（前回提案）の位置づけ

- GEM-RAGの「ユーティリティ質問 → グラフ → スペクトル分解」を医療データに転用
- 「同じEigenthemeに荷重する時間的に前後の状態 = ユーティリティ因果的」
- **変数レベルではなくテーマレベル**の「因果」

## 1. 根本的な比較: LiNGAM vs Utility Causality

### 1-1. 哲学的レベルの違い

| 軸 | LiNGAM | Utility Causality |
|---|---|---|
| **因果の定義** | 構造方程式モデル（SEM）における直接効果 | 「同じ臨床問いに答える能力」の時間的連続 |
| **識別可能性の根拠** | 非ガウス性（ICAの識別性定理） | なし（LLM + embedding + スペクトル構造に依存） |
| **粒度** | **変数レベル**（HbA1c → eGFR） | **テーマレベル**（「代謝リスク」→「腎機能悪化」） |
| **仮定の性質** | 数学的に明示的・検証可能 | 暗黙的・LLMの質問生成能力に依存 |
| **介入の概念** | 明確（do演算子に対応） | なし（情報的関連性であり介入ではない） |
| **発見の方向** | ボトムアップ（データ → 構造） | トップダウン+ボトムアップ（臨床知識 → 質問 → グラフ → 構造） |

### 1-2. 何が発見できて、何が発見できないか

```
LiNGAMが発見できてUtility Causalityが発見できないもの:
  ├─ 個別変数間の因果方向（HbA1c → eGFR vs eGFR → HbA1c）
  ├─ 因果効果の定量的サイズ（係数 b_ij）
  ├─ 数学的に保証された識別可能性
  └─ do-interventionへの直接的な回答

Utility Causalityが発見できてLiNGAMが発見できないもの:
  ├─ テーマレベルの因果連鎖（「代謝リスクモード」→「腎機能悪化モード」）
  ├─ 臨床知識を組み込んだ構造（ガイドライン・病態生理学）
  ├─ 非線形・閾値効果・相互作用の暗黙的捕捉（LLMを通じて）
  └─ フィードバックループを含む構造（DAG仮定が不要）
```

### 1-3. 「因果」の意味が異なる — これが核心

LiNGAMの「因果」: **Reichenbach-Suppes的な確率的因果**を構造方程式で表現。非ガウス性を使ってSEMのパラメータを識別。

Utility Causalityの「因果」: **「何が何の答えに役立つか」という情報的因果性**。Pierce的な記号論に近い — 「記号（データ）がどんな問い（解釈項）を生成するか」の連鎖。

**医療における重要な示唆**: 臨床医の「因果」認識は実はこの両者のハイブリッド。
- 治療方針決定時 → LiNGAM的因果（「この薬を投与すれば検査値が改善する」）
- 病態理解時 → Utility Causality的因果（「この検査パターンはあの疾患プロセスの現れだ」）

## 2. 医療データにおけるLiNGAMの課題 → Utility Causalityでの補完可能性

### 2-1. 線形性仮定

**LiNGAMの問題**: 医療データには非線形関係が豊富
- 用量反応曲線（S字型）
- 閾値効果（eGFR 60未満で急激にリスク上昇）
- U字型関係（BMIと死亡率）
- 交互作用（糖尿病 × 高血圧のリスク乗算）

**CAM-UVで一部解決**: GAM（一般化加法モデル）で非線形対応。しかしGAMは交互作用項を含まない。

**Utility Causalityでの補完**: LLMによるユーティリティ質問生成は暗黙的に非線形関係を捉える。「HbA1c 8.5%は重度の血糖コントロール不良を示す」vs「HbA1c 6.8%は軽度の上昇」— この臨床的文脈の違いがユーティリティ質問の内容に反映され、グラフ構造に織り込まれる。

### 2-2. 非ガウス性仮定

**LiNGAMの問題**: 識別可能性は非ガウス誤差に依存。しかし：
- 大規模コホートでは中心極限定理により多くの変数がガウスに近づく
- 特に健診データの血圧・BMI・脂質等は正規分布に近い
- 検査値の多くは参照範囲内に集中（切断分布 → ガウスに近い）

**実際の対処**: Kotoku et al. (2020)の大阪府健診データ研究でDirectLiNGAMが「interesting and reasonable results」を出している → 実用上は問題ないケースもあるが、理論的保証が弱まる。

**Utility Causalityでの補完**: 分布仮定が不要。グラフ構造はユーティリティ質問の意味的類似度に基づく → 変数の分布形状に依存しない。

### 2-3. DAG（非循環）仮定

**LiNGAMの問題**: 医療システムにはフィードバックループが本質的に存在
- 炎症 → 臓器障害 → さらなる炎症
- 高血糖 → 腎障害 → 糖代謝異常の悪化
- 肥満 → 運動制限 → さらなる肥満
- 精神疾患 ↔ 睡眠障害（双方向）

**VARLiNGAMでの対処**: 時間遅れによりDAG仮定を部分的に緩和（同時点はDAG、異時点間は許容）。Okuda et al. (2025)のworkflow-constrained LiNGAMも同様のアプローチ。

**Utility Causalityでの補完**: DAG仮定が不要。ユーティリティグラフは本質的に重み付き無向/有向グラフであり、循環を含められる。スペクトル分解も循環グラフに適用可能。

### 2-4. 潜在共通原因（未観測交絡）

**LiNGAMの問題**: 基本LiNGAMは「潜在共通原因なし」を仮定。RCDで緩和されるが、依然として線形仮定が残る。

**医療データの現実**: 未観測交絡は日常的
- 社会経済的地位（多くの健診データに含まれない）
- 遺伝的素因
- 生活習慣の詳細（食事内容、運動頻度）
- 服薬アドヒアランス

**Utility Causalityでの補完**: ユーティリティ質問が暗黙的に未観測要因を「問い」の形で表現しうる。例：「この検査パターンを持つ患者は、どのような生活習慣を持っている可能性があるか？」→ 未観測の生活習慣がユーティリティ質問空間に反映される。

### 2-5. 高次元性とスケーラビリティ

**LiNGAMの問題**: 変数数が増えるとDAG空間が超指数的に爆発。DirectLiNGAMはO(p³)程度だが、p=数千の検査項目では厳しい。

**Utility Causalityの問題**: ラプラシアンの固有分解もO(N³)。ノード数が患者数×変数数だと同様に厳しい。

**→ 両方とも課題だが、Utility Causalityは階層的スペクトル分解（臓器系→全身）でスケーラブルにできる可能性がある。LiNGAMはCAM-UVのような選択的因果発見で対処。**

## 3. 統合・拡張アイデア

### 3-1. 【有望度: ★★★★★】 LiNGAM → Utility Causality パイプライン（二段ロケット）

```
Phase 1: LiNGAM（変数レベル因果発見）
  Input: 横断 or 縦断の臨床データ
  Output: 変数間の因果DAG（例: HbA1c → eGFR, BP → LVH）
  
Phase 2: Utility Causality（テーマレベル因果発見）
  Input: Phase 1のDAG + 元データ
  方法: 
    - LiNGAM DAGの因果パスを「チャンク」として扱う
    - 各因果パスに対しユーティリティ質問を生成
      例: パス「HbA1c → 微量Alb → eGFR低下」
      → 質問「糖尿病性腎症の段階的進行を示すパスか？」
    - 因果パス間のユーティリティグラフを構築
    - スペクトル分解で「因果テーマ」を抽出
  Output: 因果的に根拠のあるEigenthemes
```

**利点**: LiNGAMの数学的厳密性 + Utility Causalityの臨床的解釈可能性を両立。「この因果テーマは、LiNGAMで識別された3つの因果パスから構成されている」と言える。

### 3-2. 【有望度: ★★★★☆】 Eigentheme-guided LiNGAM（次元圧縮→因果発見）

```
Phase 1: Utility Causality（テーマ抽出）
  Input: 臨床データ + LLMユーティリティ質問
  Output: Eigenthemes（潜在臨床テーマ）とそのスコア

Phase 2: LiNGAM on Eigentheme scores
  Input: 各患者のEigenthemeスコア（k次元, k << p）
  Output: テーマ間の因果DAG
  例: 「代謝リスクテーマ」→「腎機能悪化テーマ」→「心血管イベントテーマ」
```

**利点**: 
- LiNGAMの高次元問題を回避（Eigenthemeで次元圧縮）
- テーマレベルの因果DAGは臨床的に直感的
- 非ガウス性もテーマスコアでは保持されやすい（混合分布の合成→非ガウス）

**注意点**: Eigenthemeスコアが線形であるかは保証されない → CAM-UVとの組み合わせも検討

### 3-3. 【有望度: ★★★★★】 スペクトル正則化付きLiNGAM（最新研究との接続）

Oxford の M'Charrak et al. (2025) "Connected Causal Graphs for Real-World Science" がまさにこの方向：
- **Fiedler固有値（グラフラプラシアンの第2最小固有値）による正則化**をDAG学習に導入
- 「科学的データの因果グラフは連結であるべき」という帰納バイアス

**GEM-RAG的拡張**:
```
LiNGAM + スペクトル正則化:
  - DirectLiNGAMの因果順序推定にFiedler正則化を追加
  - ユーティリティグラフのラプラシアン固有構造を正則化項として利用
  
具体的には:
  argmin_B ||X - BX||² + λ₁ · Acyclicity(B) + λ₂ · SpectralReg(B, L_utility)

  L_utility: ユーティリティグラフのラプラシアン
  SpectralReg: BのDAG構造がユーティリティグラフのEigenthemeと整合するよう制約
```

**意味**: LiNGAMが見つける変数レベル因果構造が、臨床的ユーティリティのテーマ構造と矛盾しないよう制約する → **臨床知識を因果発見に数学的に組み込む方法**

### 3-4. 【有望度: ★★★★☆】 VARLiNGAM + Utility Causality for 経時データ

```
Layer 1: VARLiNGAM
  - 変数レベルの時間遅れ因果 + 同時点因果
  - 出力: B_0（同時点）, B_1, ..., B_k（時間遅れ）

Layer 2: Utility Causality（時間的）
  - VARLiNGAMの因果係数を「チャンク」の一部として利用
  - ユーティリティ質問: 「時点tでHbA1c→eGFRの因果効果が0.3である患者群は、
    どんな臨床的問いに答えられるか？」
  - 時間的ユーティリティグラフ → Eigentrajectories

Layer 3: 統合
  - VARLiNGAMの変数レベル因果をEigentrajectoryで「束ねる」
  - 出力: 「この因果軌跡テーマには、HbA1c→eGFR, BP→Alb, LDL→ABI の
    3つの変数レベル因果が含まれ、全体として『動脈硬化進展モード』を形成」
```

### 3-5. 【有望度: ★★★☆☆】 非ガウス性のスペクトル解釈

**独創的アイデア**: LiNGAMが非ガウス性を使って因果方向を識別するなら、Utility Causalityのグラフラプラシアン固有ベクトルの非ガウス性にも因果情報が含まれているのでは？

```
仮説: ユーティリティグラフのラプラシアン固有ベクトル u_k の
      各成分の分布が非ガウスであるほど、そのEigenthemeは
      因果的に「方向性を持つ」テーマである

検証方法:
  1. ユーティリティグラフを構築、スペクトル分解
  2. 各固有ベクトルの成分分布の非ガウス性を測定（尖度、ネゲントロピー等）
  3. 非ガウス性が高い固有ベクトルに対応するEigenthemeを
     LiNGAMのDAGと照合
  4. 非ガウスEigenthemeがLiNGAMの因果パスと対応するか検証
```

**もし成立すれば**: Utility CausalityにLiNGAM的な「因果方向の識別」を内在化できる。スペクトル構造から因果方向が推定できる可能性。

### 3-6. 【有望度: ★★★★★】 ワークフロー制約付きLiNGAM × ユーティリティ制約（Okuda et al. 2025の拡張）

Okuda et al. (2025)は**臨床ワークフローに基づく制約**をLiNGAMに導入（日本の健診コホート10万人超）:
- 測定プロトコルの時間順序 → DAGの許容エッジ制約
- 年次健診のタイムライン → 時間インデックス
- ブートストラップ不確実性定量化

**ユーティリティ制約との統合**:
```
既存: ワークフロー制約（測定順序・プロトコル → 許容エッジマスク）
追加: ユーティリティ制約（ユーティリティ質問の類似度 → エッジの事前確率）

統合:
  P(edge i→j) ∝ WorkflowMask(i,j) × UtilityPrior(i,j)
  
  WorkflowMask: 測定プロトコルから「i→jが可能か」
  UtilityPrior: 「iとjが同じ臨床問いに答えるならエッジが存在する確率が高い」
```

**意味**: 
- ワークフロー制約 = 「物理的にどの因果が可能か」（ハード制約）
- ユーティリティ制約 = 「臨床的にどの因果がもっともらしいか」（ソフト事前分布）
- **測定プロトコルと臨床知識の両方をLiNGAMに注入**

## 4. 横断スナップショットデータ特有の議論

### 4-1. DirectLiNGAM on 健診データ（既存研究）の分析

Kotoku et al. (2020)は大阪府健診データにDirectLiNGAMを適用:
- 性別・年齢層別に因果DAGを推定
- 「LDL → γGT」「BMI → 血圧」等の因果関係を発見
- 結果は「interesting and reasonable」だが、**臨床的に自明な関係が多い**

**問題提起**: LiNGAMは数学的に正しい因果DAGを出すが、**臨床医が「それは知っている」と言うような関係が多い**。より高次の洞察（未知のパターン、テーマレベルの構造）を得るにはLiNGAMだけでは不十分。

**→ Utility Causalityとの統合で、LiNGAMの「既知の因果」をEigenthemeとして束ね、そのテーマ間の未知の関係を発見する余地がある。**

### 4-2. 横断データでの「因果」の限界と補完

横断スナップショットでは時間的前後関係がないため、因果方向の推定がより困難：
- LiNGAM: 非ガウス性のみに依存 → 横断データでも因果方向を推定できるが、**時間的検証が不可能**
- Utility Causality: 横断データではそもそも「時間的連続」がない → **「因果」ではなく「臨床テーマ的関連構造」**として解釈するのが自然

**提案: 横断データでの「ユーティリティ関連性 (Utility Relevance)」**

> 変数Aと変数Bが「ユーティリティ関連的」であるとは、Aに基づくユーティリティ質問群とBに基づくユーティリティ質問群が、同じEigenthemeに強く荷重していることを指す。

これは因果ではないが、**LiNGAMの因果DAGでは見えない「臨床テーマ的な変数クラスタリング」**を提供する。

### 4-3. 横断データの実用的パイプライン

```
Step 1: DirectLiNGAM → 変数間因果DAG
  (例: BMI → BP → LVH, HbA1c → Cr → eGFR)

Step 2: Utility Causality → 変数間ユーティリティグラフ → Eigenthemes
  (例: Theme 1 = 「心血管リスク」, Theme 2 = 「代謝症候群」)

Step 3: DAG × Eigentheme のクロス分析
  - 各因果パスがどのEigenthemeに属するか
  - あるEigentheme内に複数の因果パスがある場合、
    それらは「同じ臨床テーマの中での因果連鎖」
  - 異なるEigenthemeを跨ぐ因果パスは
    「テーマ間の因果的橋渡し」→ 臨床的に最も興味深い発見

Step 4: 可視化
  - LiNGAM DAGのノードをEigenthemeで色分け
  - テーマ内因果（同色エッジ）とテーマ間因果（異色エッジ）の区別
  - テーマ間因果エッジ = 臨床的に新しい洞察の候補
```

## 5. 経時データ特有の議論

### 5-1. VARLiNGAM vs Utility Causality: 何が「時間的因果」か

| 側面 | VARLiNGAM | Utility Causality |
|---|---|---|
| 同時点因果 | DAG（因果順序あり） | Eigentheme内の関連（順序なし） |
| 時間遅れ因果 | B_τ 行列の非ゼロ要素 | Eigentrajectory上の時間的荷重パターン |
| フィードバック | 同時点ではNG、異時点では可 | 制約なし（循環グラフ可） |
| 動的変化 | 定常性仮定（B_τ は時間不変） | Eigenthemeの時間的変化を捉えうる |

### 5-2. 経時データの統合パイプライン

```
Step 1: VARLiNGAM → 変数レベルの時間的因果構造
  B_0: 同時点の因果DAG
  B_1, ..., B_k: 時間遅れの因果行列

Step 2: 因果インパルス応答関数（CIRF）の計算
  - VARLiNGAMから因果インパルス応答を導出
  - 「HbA1cに1単位のショック → 3年後のeGFRへの因果効果」

Step 3: CIRFベースのユーティリティ質問生成
  - 各CIRFパターンを「チャンク」として扱い、ユーティリティ質問を生成
  - 「このインパルス応答パターンはどんな臨床的介入の効果を示唆するか？」

Step 4: CIRFユーティリティグラフ → スペクトル分解
  - 出力: 「因果的インパルス応答のEigenthemes」
  - = LiNGAMで証明された因果効果が、テーマレベルでどう組織化されるか

Step 5: 臨床解釈
  - 「Eigentheme 1: 代謝介入の腎保護効果モード
     （HbA1c介入のeGFR応答 + BP介入のAlb応答 + LDL介入のABI応答が
      同じテーマに荷重）」
```

### 5-3. Longitudinal LiNGAM（Okuda et al. 2025）の拡張

Okuda et al.の手法は日本の健診コホート（107,261人、429,044人年）で実証済み。

**ユーティリティ拡張の方向**:

1. **ワークフロー制約 → ユーティリティ制約への拡張**: 
   - 現在のワークフロー制約は測定プロトコルに基づく「ハード制約」
   - ユーティリティ質問に基づく「ソフト事前分布」を追加
   - エッジの信頼度 = ブートストラップ確率 × ユーティリティ類似度

2. **Lagged total effectsの臨床的解釈**:
   - Okuda et al.はlagged total effectsのブートストラップ不確実性を計算
   - これらのtotal effectをチャンクとしてユーティリティ質問を生成
   - → 統計的に有意なtotal effectが「臨床的にどんな意味を持つか」を自動的に注釈付け

3. **介入クエリの臨床テーマ化**:
   - Okuda et al.は介入クエリを動的表現でサポート
   - ユーティリティ拡張: 「この介入はどのEigenthemeに影響するか」を計算
   - → 治療介入の効果を「テーマレベル」で評価（臓器保護効果テーマ、代謝改善テーマ等）

## 6. 大胆な仮説: 「スペクトル因果同定」は可能か？

### 6-1. 仮説の核心

LiNGAMが「非ガウス性」で因果方向を同定するように、**ユーティリティグラフの「スペクトル非対称性」で因果方向を同定できないか？**

```
着想:
  - 有向ユーティリティグラフ（時間方向あり）のラプラシアンは非対称
  - 非対称ラプラシアンの固有値は一般に複素数
  - 複素固有値の虚部 = グラフ上の「回転」= 情報の方向的フロー
  
仮説:
  虚部が大きい固有値に対応するEigenthemeは、
  因果的方向性が強いテーマである
  
  Re(λ_k) = テーマの強度（グラフ内での分散の大きさ）
  Im(λ_k) = テーマの方向性（因果的フローの強さ）
```

### 6-2. LiNGAMとの接続

もし上記仮説が成立するなら：
- **LiNGAMの因果方向 ↔ ユーティリティグラフの複素固有値の虚部**
- 両者が一致すれば: Utility Causalityに独自の因果方向同定能力が内在する
- 両者が一致しなければ: 異なる種類の「方向性」を捉えている

**検証パイプライン**:
1. VARLiNGAMで因果方向を推定
2. 有向ユーティリティグラフを構築、スペクトル分解
3. 複素固有値の虚部と、VARLiNGAMの因果方向の対応を統計的に検証

### 6-3. 横断データへの含意

もし成立すれば、**横断スナップショットデータでも**（時間方向はないが）ユーティリティ質問の非対称性（「AがBに役立つ」と「BがAに役立つ」のスコア差）から方向性を推定できる可能性がある。

これはDirectLiNGAMの非ガウス性に基づく方向同定とは**全く異なるメカニズム**で因果方向を推定することになる → 両者を組み合わせたアンサンブル因果方向推定。
